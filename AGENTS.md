# AGENTS.md

## 开发规范

本项目的实现必须以“可移植、可复用、可维护”为优先原则，避免把当前开发机器上的偶然环境当成产品能力。

### 1. 可移植优先

- 不要依赖用户机器上预装的全局工具、全局配置或全局路径。
- 优先使用项目内依赖、项目内配置、项目内可生成资源。
- 需要运行时目录时，优先放在项目可控目录或应用数据目录，不要硬编码个人目录。
- 不允许把 `/Users/...`、`C:\\...` 这类个人机器路径写入业务逻辑。
- 与平台相关的逻辑必须显式处理差异，不要默认只支持 macOS 或当前开发环境。

### 2. 配置驱动

- 所有环境差异必须通过配置项、环境变量或显式初始化流程注入。
- 默认配置应能在一个干净环境中工作，不能隐式依赖作者本机历史状态。
- 配置缺失时要给出明确错误信息，不要静默降级到不可预期行为。
- 新增能力时，优先先设计配置接口，再设计实现细节。

### 3. 复用优先

- 相同语义只保留一份实现，避免前端、后端各自维护一套不一致逻辑。
- 公共类型、协议、数据结构应放入共享层统一定义。
- 能通过通用抽象解决的问题，不要写一次性特判代码。
- 新功能要优先扩展已有模块，而不是平行再造一套新流程。

### 4. 面向真实协议实现

- 优先遵循官方文档、官方接口和真实运行时返回结构。
- 不要基于猜测实现协议字段，不要依赖未文档化能力作为主路径。
- 对外部服务返回值要做容错，但容错不能替代正确协议实现。
- 调试阶段可增加 debug 接口，最终逻辑必须回归正式接口。

### 5. UI 与能力边界清晰

- 前端负责展示、输入辅助和交互引导。
- 后端负责能力编排、协议适配和环境隔离。
- 不要把本应由后端掌握的运行时知识硬编码在前端。
- Slash command、补全、面板入口与底层执行能力要明确分层。

### 6. 可观测性

- 新增关键链路时，必须提供足够的日志、错误信息或调试入口。
- 排查问题时优先增加可观测性，不要靠猜。
- 错误提示要能帮助定位是前端、后端、配置还是第三方依赖的问题。

### 7. 改动约束

- 修改前先确认现有实现与真实行为，不要直接覆盖已有逻辑假设。
- 不要为了短期可用引入会影响后续打包、发布、迁移的临时方案。
- 临时调试代码如果保留，必须有明确用途，并且不会污染正式用户流程。
- 每次实现都要考虑后续打包成独立应用时是否仍然成立。

### 8. 默认决策

- 默认使用项目内能力，不依赖用户手工预装全局能力。
- 默认让项目在首次启动时完成最小必要初始化。
- 默认让数据结构支持后续扩展，而不是只满足当前单一场景。
- 默认优先选择更稳定、可解释、可测试的方案。

### 9. 本地依赖与运行时约束

- 默认优先使用项目内安装的 `codex`，不能把全局 `codex` 当成正式方案前提。
- 运行时依赖的 `CODEX_HOME`、缓存目录、配置目录必须可由项目或应用自身初始化，不要求用户手工准备个人环境。
- 如果某项能力依赖外部配置，必须提供项目内默认配置、初始化流程或清晰的缺失提示。
- 本地开发可使用 `.env` 注入路径，但实现逻辑不能把 `.env` 中的个人目录当成产品默认值。

### 10. MCP 集成规范

- MCP 能力必须基于官方 `app-server` 文档和正式接口实现，优先使用 `mcpServerStatus/list` 这类公开能力。
- MCP server 列表、tool 列表、auth 状态、resource 信息应来自后端真实返回，不能在前端硬编码。
- `/mcp` 输入建议属于前端交互层，但建议数据必须由后端按真实协议提供。
- 如果项目内需要内置 MCP server，应通过项目初始化命令或项目内配置生成，而不是要求用户预先全局安装。
- 调试 MCP 时可临时保留原始状态查看接口，但正式交互必须走统一的业务接口。

### 11. 状态一致性规范

- 同一语义状态只能有一个可信来源，界面展示状态与实际执行状态必须共享同一份数据。
- 线程选中状态、当前详情、发送目标、发送中状态必须保持一致，不能出现“显示在线程 A、实际发到线程 B”的情况。
- 失败恢复逻辑应尽量自动化，但不能偷偷创建新资源掩盖真实错误。
- 任何自动恢复、自动重试都必须以不改变用户预期目标为前提。

### 12. 调试与正式能力的边界

- Debug 接口可以保留，但命名、用途和输出必须明确，不得与正式业务接口混用。
- 前端正式功能不能长期依赖 debug 接口返回。
- 为排查协议问题新增的日志与调试入口，在问题稳定后应继续保留最小必要观测能力，而不是完全移除。
- 临时兜底方案如果偏离官方接口，必须标注为过渡实现，并尽快回归正式协议。
