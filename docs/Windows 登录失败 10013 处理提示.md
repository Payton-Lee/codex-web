# Windows 登录失败 10013 处理提示

## 适用场景

当应用内点击 `使用 ChatGPT 登录` 后，出现类似错误时，可参考本说明：

- `failed to start login server`
- `os error 10013`
- `以一种访问权限不允许的方式做了一个访问套接字的尝试`

这类错误通常不是前端按钮失效，也不是项目本身没有启动，而是 `Codex` 在本机启动登录回调服务时，绑定 `localhost:1455` 失败。

## 当前项目中的已知表现

在本项目中，后端会尝试调用本地 `codex app-server` 完成 ChatGPT 登录流程。  
登录时，`Codex` 会在本机开启一个本地回调端口用于接收浏览器登录结果。当前已知该端口通常为：

- `localhost:1455`

如果 Windows 把这个端口所在区间标记为系统保留端口，或者被虚拟化网络组件限制，就会出现 `10013`。

## 是否适合参考外部文章中的做法

可以参考思路，但不建议把文章中的系统级处理步骤原样作为应用默认提示。

原因：

- 某些做法会影响 `Hyper-V`、`WSL`、`Docker Desktop`、`WinNAT` 或 VPN
- 某些命令需要管理员权限
- 某些处理会影响整台机器的网络与虚拟化环境
- 对普通用户来说，直接执行系统级命令风险偏高

因此，应用内提示建议遵循：

1. 先给出低风险解释
2. 再给出低风险替代方案
3. 最后才列出高级系统排查建议

## 推荐提示文案

建议在应用内将这类错误提示为：

> 当前 Windows 阻止了 `localhost:1455`，Codex 无法启动 ChatGPT 登录回调服务。  
> 这通常与系统保留端口、Hyper-V、WSL、Docker Desktop 或 VPN 有关。  
> 建议优先保持当前登录态，或改用 API key 登录。  
> 如果必须重新登录，请按下方“高级排查”步骤检查系统端口保留范围。

## 低风险处理建议

### 1. 不要轻易退出当前登录态

如果当前已经登录成功，优先继续使用，不要反复执行“退出登录 -> 重新登录”。

这是最稳妥的方式，因为该问题只影响“重新拉起本地登录回调服务”，不一定影响已建立的登录态。

### 2. 改用 API key 登录

如果业务允许，可直接改用 API key 登录，以绕开本地 `localhost:1455` 的登录回调端口。

### 3. 先检查是否命中系统排除端口范围

在 PowerShell 中执行：

```powershell
netsh interface ipv4 show excludedportrange protocol=tcp
netsh interface ipv6 show excludedportrange protocol=tcp
```

如果输出中包含 `1455` 所在区间，例如：

```text
1444 - 1543
```

就说明本机系统层面已经禁止该端口被普通进程绑定。

## 高级排查建议

只有在确实需要重新使用 ChatGPT 登录，且能接受系统级操作风险时，才建议继续执行。

重点排查对象：

- `Hyper-V`
- `WSL`
- `Docker Desktop`
- `WinNAT`
- VPN 或安全软件

这些组件可能会让 Windows 保留一段本地端口区间，导致 `1455` 无法使用。

### 可选处理：临时停用相关网络/虚拟化服务

如果你确认当前机器上的 `10013` 与 `WinNAT`、`HNS`、虚拟机管理服务有关，并且能够接受暂时影响 Docker / WSL / Hyper-V / 虚拟网络能力，可以尝试以下方案。

请注意：

- 需要管理员权限
- 会影响系统当前的虚拟化网络能力
- 不适合默认推荐给普通用户
- 更适合作为“临时重新登录”的排障步骤

管理员 PowerShell 或管理员终端中执行：

```powershell
net stop winnat
net stop hns
net stop vmms
```

完成登录后，如需恢复，再执行：

```powershell
net start winnat
net start hns
net start vmms
```

### 适用判断

这个方案更适合下面这些情况：

- 本机必须重新走 ChatGPT 登录
- 已确认 `1455` 命中了系统排除端口范围
- 机器上启用了 Docker Desktop、WSL、Hyper-V 或其他虚拟化网络能力
- 能接受短时间停用这些相关服务

### 不建议直接用于默认提示

即使该方案在部分机器上有效，也不建议在应用主提示中直接显示为第一处理方案。更推荐的顺序仍然是：

1. 不要轻易退出当前登录态
2. 改用 API key 登录
3. 再查看本节高级处理步骤

## 不建议直接作为默认提示的内容

以下类型内容不建议直接放进应用主提示区：

- 直接要求用户停系统服务
- 直接要求用户删除端口排除范围
- 直接要求用户重置网络组件
- 直接要求用户关闭 Hyper-V / Docker / WSL

这些操作更适合放在“高级排查”或“管理员指导”中，而不是普通用户默认提示。

2. 将本文件作为详细说明文档保留在项目内
