# 线程历史中期改造设计

## 目标

将线程与消息历史的主数据源统一到用户真实的 `CODEX_HOME`（默认 `~/.codex`），使 `codex-cli`、`codex-app`、`codex-webui` 共享同一套线程事实数据。

本次设计遵循以下原则：

- `state_*.sqlite` 负责线程索引与摘要。
- `sessions/**/*.jsonl` rollout 负责原始事件事实。
- `apps/server` 自身 SQLite 只做归一化缓存、兼容读取模型、session 辅助信息。
- 前端不直接感知 rollout 原始结构差异，只消费稳定的 `ThreadSummary` / `ThreadDetail` / `ThreadHistoryPage`。

## 为什么不能只直接读 rollout

rollout jsonl 是 append-only 事件流，真实且完整，但存在两个问题：

1. 事件结构未来可能演进。
2. 前端直接依赖原始事件会让 UI 协议和运行时格式耦合过深。

因此本项目采用三层模型：

1. `.codex/state_*.sqlite`
   用于线程目录、标题、cwd、更新时间、rollout path 等索引信息。
2. `.codex/sessions/**/*.jsonl`
   用于消息、推理、命令、文件变更等事实事件。
3. `apps/server` 的业务 SQLite
   用于归一化后的 turns/items/changes 和 ingest checkpoint。

## 解析策略

### 线程索引

后端优先读取 `~/.codex/state_*.sqlite` 的最新代际文件：

- 读取 `threads` 表
- 按 `updated_at_ms` 排序
- 缓存到 `codex_source_threads`

### rollout ingest

当前第一版采用“线程级重建”策略：

1. 读取 rollout 文件的 `size + mtime`
2. 与 `codex_thread_ingest` 记录比较
3. 若未变化，直接读取缓存
4. 若变化，重新解析该 thread 的整个 rollout 文件
5. 删除旧的 turns/items/changes，写入新结果

这个实现的优点是：

- 逻辑稳定
- 容错简单
- 便于先把结构做正确

后续可优化为增量 ingest，但不影响当前前后端协议。

## 向前兼容策略

如果未来 rollout 结构变化，不应直接改前端，而应优先修改：

1. rollout parser
2. ingest 兼容逻辑
3. 必要时重建归一化缓存

只要 `ThreadDetail` / `ThreadHistoryPage` 结构稳定，前端就不需要跟随底层格式一起震荡。

## 当前对外接口

### 已切换到 `.codex` 主读取源

- `GET /api/threads`
- `GET /api/threads/:threadId`

### 新增历史分页接口

- `GET /api/threads/:threadId/history?beforeTurnId=...&limit=...`

返回：

- `turns`
- `hasMore`
- `nextBeforeTurnId`

这为后续前端“向上滚动加载更多历史”提供了正式接口。

## 当前仍保留的兼容/过渡点

- `POST /api/threads/:threadId/turns` 发送消息仍然走 app-server 正式能力
- 当 `.codex` 侧暂时读不到某个 thread 详情时，`GET /api/threads/:threadId` 仍保留 app-server fallback
- `loaded threads` 仍由 app-server 给出，因为它表达的是运行时加载状态，而不是历史事实状态

## 后续建议

1. 前端 `ChatView` 改造成按 turn 分页加载
2. 进入线程时先拉最近一页历史，再向上滚动加载更多
3. `pendingPrompt` / `liveDeltas` 与分页历史在 store 层合并
4. 后续把 diff 预览也逐步切到缓存层读取，减少对 `thread/read` 的依赖
5. 在 parser 中增加未知事件计数与观测日志，便于追踪 rollout 新结构
