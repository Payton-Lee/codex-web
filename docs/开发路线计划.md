# 开发路线计划

## 一、目标

当前项目的整体开发路线明确为三段式推进：

1. 先把官方 `app-server` 能力尽可能完整地封装到后端
2. 再由前端逐步接入这些后端业务接口与实时事件
3. 最后以 Electron 作为桌面壳进行跨平台桌面化封装与打包

这份文档用于固定后续开发节奏，避免在功能尚未稳定时过早把重心转到桌面壳细节，或者在桌面适配过程中反向污染业务层实现。

## 二、总体原则

### 1. 后端优先

- 后端是能力中枢，优先承接官方协议、运行时编排、配置与环境隔离。
- 前端不直接理解或猜测官方 `app-server` 细节。
- 任何新能力都优先先落到 `apps/server`，形成稳定的业务接口后，再交给前端消费。

### 2. 前端按业务接口接入

- 前端只接后端统一暴露的 `/api` 和 WebSocket 事件。
- 前端避免直接耦合官方 JSON-RPC 细节。
- 前端重点负责交互体验、状态展示、错误提示、调试入口和恢复路径。

### 3. Electron 放在第三阶段

- Electron 不是主业务实现层，而是桌面运行时与打包交付层。
- 平时主要以 `web + server` 模式开发业务。
- 在功能稳定后，再进入 Electron 集成、路径处理、跨平台差异和安装包验证。

## 三、阶段划分

## 阶段一：后端接口完整封装

### 目标

围绕官方 `app-server` README，把当前后端尚未封装的高价值能力逐步补齐，统一暴露为本项目可消费的 HTTP / WebSocket 业务接口。

### 范围

- `apps/server`
- `packages/shared`

### 重点任务

#### 1. 补齐后端未封装的官方高价值接口

优先顺序建议如下：

- `thread/compact/start`
- `thread/rollback`
- `thread/backgroundTerminals/clean`
- `thread/unsubscribe`
- `thread/metadata/update`
- `thread/shellCommand`
- `thread/inject_items`
- `memory/reset`
- `thread/memoryMode/set`
- `tool/requestUserInput`
- `app/list`
- `plugin/read`
- `config/mcpServer/reload`

第二优先级：

- `command/exec`
- `command/exec/write`
- `command/exec/resize`
- `command/exec/terminate`
- `fs/readFile`
- `fs/writeFile`
- `fs/createDirectory`
- `fs/getMetadata`
- `fs/readDirectory`
- `fs/remove`
- `fs/copy`
- `fs/watch`
- `fs/unwatch`

第三优先级：

- `thread/realtime/start`
- `thread/realtime/appendAudio`
- `thread/realtime/appendText`
- `thread/realtime/stop`
- `marketplace/add`
- `plugin/install`
- `plugin/uninstall`
- `skills/config/write`
- `mcpServer/oauth/login`
- `windowsSandbox/setupStart`
- `feedback/upload`
- `config/read`
- `config/value/write`
- `config/batchWrite`
- `configRequirements/read`
- `experimentalFeature/enablement/set`
- `externalAgentConfig/detect`
- `externalAgentConfig/import`

#### 2. 补齐共享类型与前端事件协议

在新增后端接口的同时，同步维护：

- `packages/shared/src/index.ts`

需要确保：

- 请求参数与返回结构统一
- 新通知事件有共享类型定义
- 线程、审批、文件系统、命令执行、实时会话的状态结构可扩展

#### 3. 补齐可观测性

每新增一条关键链路，都需要补齐：

- 请求日志
- 错误日志
- 必要的调试接口或调试字段
- 明确错误提示

### 阶段一验收标准

- 高价值官方接口都能通过后端统一访问
- 后端不再依赖前端自行理解 `app-server` 协议细节
- 每条新链路都有基本日志
- 共享类型与实际行为一致

## 阶段二：前端逐步接入后端能力

### 目标

在后端接口稳定的前提下，由前端逐项接入业务能力、实时事件和调试入口，形成完整的用户可见功能。

### 范围

- `apps/web`
- `packages/shared`

### 重点任务

#### 1. 先补 API 封装

按后端接口清单补齐 `apps/web/src/api.ts`：

- 线程重命名
- 线程归档 / 取消归档
- 线程 fork
- steer
- review
- 登录取消
- MCP resource read / tool call
- 后续的 compact / rollback / shellCommand / command exec / fs 等接口

#### 2. 再补状态层接入

按功能补齐 `apps/web/src/store.ts`：

- 新接口调用
- 成功后状态刷新
- 错误处理
- 恢复逻辑
- 事件驱动更新

#### 3. 补齐后端事件消费

优先补这些实时事件：

- `thread.status`
- `thread.archived`
- `thread.unarchived`
- `thread.named`

后续扩展：

- `command/exec/outputDelta`
- `fs/changed`
- `skills/changed`
- `thread/realtime/*`
- `thread/closed`

#### 4. 做登录与调试体验

优先把可观测性补齐：

- 登录 loading / error 提示
- 登录返回 `authUrl` 时的 fallback 展示
- 浏览器未弹出时的手动打开入口
- MCP / command / fs / plugin 等高级能力的调试面板

#### 5. 做线程与能力管理交互

建议优先顺序：

1. 线程重命名
2. 线程归档 / 取消归档
3. steer / review
4. fork
5. MCP 直调
6. compact / rollback
7. command exec / fs

### 阶段二验收标准

- 前端已把高价值后端能力接入成实际可用功能
- 线程状态、详情、发送目标与交互一致
- 登录与错误路径可见、可解释、可恢复
- 前端不直接依赖官方 JSON-RPC 细节

## 阶段三：Electron 桌面化与安装包交付

### 目标

在 `web + server` 业务能力稳定后，由 Electron 负责桌面运行时、进程编排、系统交互与安装包产出。

### 范围

- `apps/desktop`
- 打包配置
- 跨平台资源准备

### 重点任务

#### 1. 桌面运行时稳定化

- 启动本地 `server`
- 注入桌面环境变量
- 使用项目内打包的 `codex`
- 统一日志目录、数据目录、`CODEX_HOME`

#### 2. 桌面调试能力

- 主进程启动日志
- 子进程 stdout / stderr 日志
- 关键错误弹窗
- 桌面模式健康检查

#### 3. 打包方案

- Windows：`nsis`
- macOS：`dmg`

#### 4. 资源可移植性

- 不依赖全局 `codex`
- 不依赖个人路径
- 不依赖仓库 cwd
- 运行时目录统一由应用自身初始化

#### 5. 跨平台验证

Windows 重点验证：

- 首次启动
- 登录
- 线程创建 / 恢复
- prompt / interrupt
- 审批
- diff
- 退出与进程回收

macOS 重点验证：

- 应用启动
- 本地服务拉起
- 浏览器登录
- 资源路径
- dmg 安装与启动

### 阶段三验收标准

- `win-unpacked` 和 Windows 安装包可正常运行
- macOS `dmg` 可启动并完成主链路验证
- 桌面版不依赖开发机特定环境
- 主业务仍然保持 `web + server` 架构稳定

## 四、推荐开发模式

### 日常功能开发

默认优先：

- `apps/server`
- `apps/web`

也就是：

- 业务逻辑优先按 Web 应用方式开发
- 桌面壳不参与大多数日常功能迭代

### 阶段性桌面验证

在以下场景切到 Electron：

- 登录与系统浏览器
- 本地进程拉起
- 路径与资源加载
- 桌面打包
- 平台差异

### 结论

开发方式不是“全程 Electron-first”，而是：

- 业务能力以 `web + server` 为主线开发
- Electron 作为桌面封装层阶段性验证与交付

## 五、推荐实施顺序

建议按下面顺序推进：

1. 继续补齐后端尚未封装的高价值官方接口
2. 同步维护共享类型
3. 为每条关键链路补日志和错误提示
4. 前端补 `api.ts`
5. 前端补 `store` 与事件消费
6. 完成线程管理与登录调试交互
7. 再做 Electron 桌面联调
8. 最后验证 Windows 安装包与 macOS dmg

## 六、当前阶段建议

结合当前项目状态，下一步建议继续停留在阶段一和阶段二之间，不要过早把主要精力切到打包层。

当前优先事项建议为：

1. 继续补后端接口封装
2. 补“后端已接、前端未接”的能力
3. 补登录调试链路与线程管理交互
4. 完成后再做新一轮 Electron 验证
