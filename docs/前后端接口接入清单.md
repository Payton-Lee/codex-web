# 前后端接口接入清单

## 目的

这份清单用于对照当前项目里：

- 后端 `apps/server` 已提供的 `/api` 接口
- 前端 `apps/web` 已封装的 `api.ts`
- 前端页面与状态层实际接入的交互能力

重点关注两类缺口：

1. 后端已经接好，但前端连 API 封装都还没有
2. 前端已经有 API 封装，但页面和状态层还没有真正接入

## 对照范围

- 后端路由：`apps/server/src/app.ts`
- 前端 API：`apps/web/src/api.ts`
- 前端状态层：`apps/web/src/store.ts`
- 前端页面组件：`apps/web/src/components/*`
- 共享事件：`packages/shared/src/index.ts`

## 一、后端已接，前端还没有 API 封装

这些接口已经在 `server` 暴露了 `/api` 路由，但 `apps/web/src/api.ts` 里还没有对应方法。

### 账号与调试

- `POST /api/account/login/cancel`
  用途：取消进行中的登录流程。
  现状：后端已接，前端未封装，登录流程也没有取消 UI。

- `GET /api/health`
  用途：健康检查。
  现状：主要用于桌面启动联调，前端未封装。

- `GET /api/debug/mcp-status`
  用途：调试 MCP 原始状态。
  现状：后端已接，前端没有调试入口。

### 线程高级操作

- `POST /api/threads/:threadId/fork`
  用途：从当前线程分叉新线程。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/steer`
  用途：在运行中追加 steer 指令。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/review`
  用途：启动 review 模式。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/archive`
  用途：归档线程。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/unarchive`
  用途：取消归档线程。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/name`
  用途：修改线程名称。
  现状：后端已接，前端未封装。

### MCP 线程能力

- `POST /api/threads/:threadId/mcp/resource/read`
  用途：读取 MCP resource。
  现状：后端已接，前端未封装。

- `POST /api/threads/:threadId/mcp/tool/call`
  用途：直接调用 MCP tool。
  现状：后端已接，前端未封装。

## 二、前端有 API 封装，但页面/状态层还没有真正用起来

这些方法已经在 `apps/web/src/api.ts` 里存在，但当前页面或 `store` 没有实质接入，或者只在很窄的场景里用到。

### 已封装但当前未使用

- `api.account()`
  现状：前端未直接调用。
  说明：当前账号状态主要依赖 `bootstrap` 和 `account.updated` 事件。

- `api.workspaces()`
  现状：前端未直接调用。
  说明：当前工作区主要依赖 `bootstrap`、`addWorkspace`、`selectWorkspace`。

- `api.approvals()`
  现状：前端未直接调用。
  说明：审批列表目前主要靠 `bootstrap` 快照和 `approval.created` / `approval.resolved` 事件。

- `api.settings()`
  现状：前端未直接调用。
  说明：设置目前主要来自 `bootstrap.snapshot.settings` 和 `server.status`。

## 三、后端通知已定义，但前端还没有消费

这些事件已经在共享类型 `FrontendEvent` 中定义，且后端已经具备对应广播语义，但前端 `store.handleEvent` 还没有处理。

- `thread.status`
  风险：线程状态变化后，侧边栏展示可能依赖下一次全量刷新，不能即时更新。

- `thread.archived`
  风险：前端无法即时把线程从当前列表移除或标记为已归档。

- `thread.unarchived`
  风险：取消归档后不能即时恢复显示。

- `thread.named`
  风险：线程改名后，侧边栏与详情区可能不能即时反映最新名称。

## 四、当前最值得优先接入的前端缺口

如果按“用户可见价值”和“对后续桌面化最有帮助”排序，建议优先做这几项：

### P1：线程管理能力

- 线程重命名：`POST /api/threads/:threadId/name`
- 线程归档 / 取消归档：`archive` / `unarchive`
- 线程状态事件消费：`thread.status`、`thread.archived`、`thread.unarchived`、`thread.named`

原因：

- 这是最直接的用户可见能力
- 和线程列表、当前选中状态强相关
- 能顺手把状态一致性做扎实

### P1：登录调试与恢复能力

- `POST /api/account/login/cancel`
- 前端登录状态提示、错误提示、手动打开 `authUrl` 的 fallback

原因：

- 当前登录“点了没反应”的可观测性不够
- 桌面环境下尤其需要失败提示和恢复入口

### P2：运行中 steer / review

- `POST /api/threads/:threadId/steer`
- `POST /api/threads/:threadId/review`

原因：

- 已经有很高的产品价值
- 后端已接，前端接入成本主要是交互设计

### P3：线程 fork 与 MCP 直调

- `POST /api/threads/:threadId/fork`
- `POST /api/threads/:threadId/mcp/resource/read`
- `POST /api/threads/:threadId/mcp/tool/call`

原因：

- 能力很强，但相对更偏高级用户与后续增强

## 五、建议的实施顺序

推荐按下面顺序推进：

1. 先补登录调试链路
2. 再补线程重命名、归档、取消归档
3. 再补 `thread.*` 实时事件消费
4. 然后接 `steer` / `review`
5. 最后接 `fork` 和 MCP 直调

## 六、备注

- `GET /api/debug/mcp-status` 建议保留为调试入口，不建议长期作为正式 UI 主路径
- `api.account()`、`api.workspaces()`、`api.approvals()`、`api.settings()` 当前虽未直接使用，但不属于错误实现，更像是“保留的显式刷新入口”
- 当前 `store` 已大量依赖快照与 WebSocket 事件，因此后续新增接口时要优先考虑“调用成功后是否也有实时事件补齐状态”
