# 桌面安装包落地方案

## 1. 文档目的

本文档用于指导当前 `codex-web` 项目在尽量少改动现有前后端逻辑的前提下，落地桌面应用版本，并产出：

- Windows 安装包：`exe` / `nsis`
- macOS 安装包：`dmg`

本文档既是方案说明，也是后续实施时的施工清单。推荐后续所有桌面化改造都以本文档为准逐步推进，并在完成每个阶段后更新状态。

## 2. 目标与约束

### 2.1 目标

- 保留当前本地 `server` + `web` 的整体架构。
- 前端交互逻辑尽量不改，优先复用已有 `/api` 和 `/ws` 调用。
- 支持打包成 Windows 安装包和 macOS dmg。
- 桌面版仍然通过本地 `codex app-server` 运行核心能力。
- 为后续签名、自动更新、原生能力扩展预留空间。

### 2.2 约束

- 不在第一版中将前端重写为 Electron IPC 驱动模式。
- 不在第一版中推翻现有后端 HTTP/WebSocket 结构。
- 不在第一版中做自动更新。
- 不在第一版中处理复杂的多窗口、多实例协同问题。

## 3. 当前项目结构与运行方式

当前项目本质上是一个本地 Web 应用：

- `apps/server`
  - 本地 Node 服务
  - 提供 `/api` 和 `/ws`
  - 管理 `codex app-server`
  - 在生产模式下托管 `apps/web/dist`
- `apps/web`
  - React + Vite 前端
  - 通过相对路径访问后端，例如 `/api/bootstrap`
- `packages/shared`
  - 前后端共享类型

当前链路可以概括为：

```text
浏览器
  -> 本地 server
  -> codex app-server
```

## 4. 推荐方案

### 4.1 方案结论

推荐使用 `Electron` 作为桌面应用外壳。

采用方式不是把现有前端业务逻辑迁移为 Electron IPC 架构，而是：

- Electron 负责应用窗口、安装包、生命周期管理
- Electron 启动本地 `apps/server`
- `apps/server` 保持当前职责不变
- 前端继续通过 `/api` 和 `/ws` 与本地服务通信
- Electron 窗口直接加载本地服务地址

### 4.2 目标架构

```text
Electron
  -> 启动本地 server 子进程
  -> BrowserWindow 加载本地 HTTP 地址

server
  -> 提供静态前端资源
  -> 提供 /api 与 /ws
  -> 管理 codex app-server
```

### 4.3 为什么选择 Electron

- 对现有前端改动最小。
- 现有 API 层可以继续使用。
- 现有后端结构可以保留。
- Windows 和 macOS 的安装包生态成熟。
- 后续扩展原生能力时路径清晰，例如托盘、自动启动、系统菜单、深链等。

## 5. 为什么不建议第一版重写为 IPC 架构

如果第一版直接改成：

- 前端通过 `preload` 暴露方法
- 前端不再走 `/api`
- Electron main 直接承接业务调用

会带来这些问题：

- 前端请求层要整体重写
- WebSocket 或流式事件模型要重做
- 当前调试方式被改变
- 后端和桌面端职责边界变得不清晰

因此第一版不建议这样做。第一版更适合把 Electron 当作容器，而不是重构业务中枢。

## 6. 第一版的边界

### 6.1 本期纳入范围

- 新增桌面端应用壳
- 支持本地启动与关闭 `server`
- 支持加载本地 Web UI
- 支持 Windows 安装包
- 支持 macOS dmg
- 支持桌面环境下的数据目录、日志目录、静态资源目录配置

### 6.2 本期暂不纳入范围

- 自动更新
- 系统托盘
- 开机自启
- 深链登录回调优化
- 多实例竞争治理
- 将所有 API 迁移到 Electron IPC
- 将 `codex` CLI 内置进安装包

## 7. 目录与代码组织建议

建议新增桌面端目录：

```text
apps/
  server/
  web/
  desktop/
packages/
  shared/
docs/
```

建议 `apps/desktop` 结构如下：

```text
apps/desktop/
  package.json
  tsconfig.json
  src/
    main.ts
    preload.ts
```

各文件职责建议：

- `main.ts`
  - 启动 Electron 主进程
  - 拉起本地 `server`
  - 等待健康检查通过
  - 创建窗口
  - 监听退出事件并清理子进程
- `preload.ts`
  - 第一版仅做安全桥接预留
  - 可以先保持极简，不承接业务调用

## 8. 最小改动原则

### 8.1 前端原则

前端原则是“尽量不改业务交互逻辑”。

当前前端 API 层已经使用相对路径：

- `/api/...`
- `/ws`

只要 Electron 最终加载的是本地 `server` 的 HTTP 地址，这一层逻辑可以继续保留，不需要大改。

### 8.2 后端原则

后端原则是“保留行为，仅增强打包兼容性”。

需要优先改造的是：

- 静态资源目录解析
- 数据目录解析
- 日志目录解析
- 优雅退出
- 生产模式路径定位

## 9. 详细实施步骤

### 阶段 1：补齐后端的桌面运行能力

状态：已完成第一批关键改造

#### 目标

让 `server` 脱离“必须在仓库根目录运行”的假设，能在打包后的资源目录中稳定工作。

#### 需要修改的点

1. 支持通过环境变量指定静态资源目录

建议新增例如：

- `CODEX_WEB_STATIC_DIR`

行为建议：

- 若设置该环境变量，则 `server` 使用该绝对路径作为前端静态资源目录
- 若未设置，则继续使用当前默认逻辑

2. 支持通过环境变量指定数据目录和日志目录

当前已有：

- `DATA_DIR`
- `AUDIT_LOG_DIR`

建议补充要求：

- 支持 Electron 在启动时传入绝对路径
- 生产模式下默认写入 Electron 用户数据目录，而不是仓库相对路径

3. 检查所有依赖 `process.cwd()` 的逻辑

重点确认：

- 静态资源查找
- 配置文件查找
- 数据文件路径
- 日志路径

凡是依赖仓库根目录的逻辑，都要补一层“桌面模式兼容”。

#### 涉及文件

- `apps/server/src/app.ts`
- `apps/server/src/config.ts`
- 如有需要，补充 `apps/server/src/...` 其他路径处理文件

#### 验收标准

- 在仓库外部目录运行打包后的 `server` 也能找到静态资源
- 日志和 SQLite 数据能落到指定绝对路径
- 不影响现有 Web 模式开发

### 阶段 2：新增 Electron 桌面壳

状态：已开始

#### 目标

新增桌面端应用壳，但不改业务主链路。

#### 需要完成的能力

1. 新建 `apps/desktop`
2. 接入 `electron`
3. 编写主进程 `main.ts`
4. 编写基础 `preload.ts`
5. 区分开发模式与生产模式

#### 主进程职责设计

主进程启动时应完成以下步骤：

1. 计算运行目录与资源目录
2. 生成本地用户数据目录
3. 拉起 `server` 子进程
4. 注入运行环境变量
5. 轮询 `/api/health`
6. 健康检查通过后创建窗口
7. 加载本地服务 URL
8. 应用退出时回收 `server`

#### 推荐注入的环境变量

- `HOST=127.0.0.1`
- `SERVER_PORT=<固定或计算出的端口>`
- `WEB_PORT=<可选，仅开发模式需要>`
- `CODEX_WEB_STATIC_DIR=<web dist 目录>`
- `DATA_DIR=<用户目录下 data>`
- `AUDIT_LOG_DIR=<用户目录下 logs>`
- `CODEX_HOME_DIR=<用户目录下 codex-home>`

#### 第一版窗口加载策略

生产模式建议直接加载：

- `http://127.0.0.1:<serverPort>`

不建议第一版使用：

- `file://`
- 自定义协议

#### 涉及文件

- `apps/desktop/package.json`
- `apps/desktop/tsconfig.json`
- `apps/desktop/src/main.ts`
- `apps/desktop/src/preload.ts`

#### 验收标准

- 能从 Electron 成功打开桌面窗口
- 能自动拉起本地 `server`
- 能正常打开首页并完成基础交互

### 阶段 3：开发模式与生产模式分离

状态：未开始

#### 目标

让桌面端既能本地开发调试，也能生产打包。

#### 建议模式

1. 开发模式

Electron 加载开发环境地址，例如：

- `http://127.0.0.1:10000`

此时你可以继续使用原有开发命令。

2. 生产模式

Electron 启动内置 `server`

由 `server` 提供：

- 静态页面
- `/api`
- `/ws`

#### 说明

推荐将“开发模式”和“生产模式”严格分开，避免：

- 一套路径同时兼容所有情况
- 调试成本过高
- 构建后行为和开发时行为不一致

#### 验收标准

- 开发模式可以正常加载本地开发页面
- 生产模式可以正常加载打包后的页面

### 阶段 4：打包配置

状态：未开始

#### 目标

接入安装包构建能力。

#### 推荐工具

- `electron-builder`

原因：

- Windows `nsis` 支持成熟
- macOS `dmg` 支持成熟
- 社区使用广泛
- 配置直观

#### 需要配置的内容

1. 应用元信息

- `appId`
- `productName`
- `artifactName`

2. 打包内容

需要确保这些内容进入安装包：

- Electron 主进程产物
- `apps/server/dist`
- `apps/web/dist`
- 必要的配置模板

3. 平台目标

Windows：

- `nsis`

macOS：

- `dmg`

#### 验收标准

- 能产出 Windows 安装包
- 能产出 macOS dmg
- 安装后应用可正常启动

### 阶段 5：验证与回归测试

状态：未开始

#### 目标

确保桌面版不是“能打开”，而是“核心能力可用”。

#### 必测项

1. 桌面应用启动
2. 首页加载
3. 登录流程
4. 工作区选择
5. 新建线程
6. 发送 prompt
7. 流式内容展示
8. 审批处理
9. diff 预览
10. 应用退出时子进程清理

#### 建议增加的桌面专项测试

1. `codex` 命令不存在时的提示
2. 端口被占用时的提示
3. `server` 启动失败时的提示
4. 日志目录不可写时的提示

## 10. 需要新增的脚本建议

建议在根目录逐步补充这些脚本：

- `npm run desktop:dev`
- `npm run desktop:build`
- `npm run dist:win`

## 11. 关键设计决策

### 11.1 是否使用 Electron

结论：是。

原因：

- 与现有项目结构最匹配
- 对前端影响最小
- 安装包能力成熟

### 11.2 是否重写前端为 IPC 模式

结论：第一版否。

原因：

- 成本高
- 对现有逻辑侵入大
- 对当前目标不是必要条件

### 11.3 是否把 `codex` CLI 打包进安装包

结论：第一版不建议。

推荐第一版继续依赖用户本机环境，并在启动时检测：

- `codex` 是否可执行
- 如果不可执行，则给出明确提示

后续如果需要进一步提升用户体验，再评估内置或首次启动自动安装策略。

## 12. 风险与应对

### 风险 1：`codex` CLI 缺失

影响：

- 应用能启动，但核心能力不可用

应对：

- 启动时检测
- 在桌面 UI 中提供明确错误提示

### 风险 2：打包后路径失效

影响：

- 找不到 `web dist`
- 找不到日志目录
- 找不到数据目录

应对：

- 所有关键路径改为显式环境变量注入
- 避免依赖仓库工作目录

### 风险 3：固定端口冲突

影响：

- 桌面应用无法启动

应对：

- 第一版先保留固定端口并提示错误
- 第二版再评估自动探测空闲端口

### 风险 4：macOS 分发限制

影响：

- 未签名应用可能被系统拦截

应对：

- 第一版先支持本地验证打包
- 正式分发前补齐签名与 notarization

## 13. 推荐的实施顺序

推荐按以下顺序推进，不建议并行大改：

1. 先补 `server` 的静态路径和数据路径兼容
2. 新增 `apps/desktop`
3. 跑通桌面开发模式
4. 跑通桌面生产模式
5. 跑通 Windows 安装包
6. 再处理 macOS dmg
7. 最后再考虑签名、自动更新和原生增强

## 14. 执行时的工作方式建议

后续实施建议遵循以下原则：

1. 一次只做一个阶段
2. 每个阶段结束后更新本文档状态
3. 每次代码改动都回指本文档的对应章节
4. 避免在桌面化过程中顺手重构无关业务逻辑

建议在后续每个阶段标题下增加状态标记，例如：

- 状态：未开始
- 状态：进行中
- 状态：已完成

## 15. 下一步建议

建议下一步正式进入实施时，优先做以下内容：

1. 在 `server` 中支持 `CODEX_WEB_STATIC_DIR`
2. 在 `server` 中理顺绝对路径的数据目录与日志目录
3. 新建 `apps/desktop`
4. 接入 Electron 最小可运行主进程

完成这 4 步后，就可以开始在本机验证“桌面壳 + 本地服务”的主链路是否跑通。
